## FLAGSレジスタ
1. ゼロフラグ
cmp %a, %b
%b - %aが、ゼロだった時、1になる。

2. キャリーフラグ
・足し算で桁上がりが発生した時。
・小さい数から大きい数を引いて、桁借りが発生した時。
jb とかで使う。


3. パリティフラグ
計算結果の一番下の1バイトのなかに、2が偶数個あった時。

4. サインフラグ
最上位ビットが1になった時。
コンピュータの世界では、一番左のビットが1なら「マイナス」というルールがある。
つまり、計算結果がマイナスになったよ。というフラグ。


他にも制御フラグがある。

## 割り込み許可フラグ(IF Interrupt Enable Flag)

このスイッチが1だとキーボードやタイマーからの割り込みを受け付ける。
0にしたら、一切無視する。
OSでは、重要な処理をするとき、この割り込み許可フラグを0にする命令を
・cli (clear interrupt)を実行してフラグを切り替える。
・sli (set interrupt)を実行して1に戻す。


## Makefileのddコマンドとかその辺り。

Data DefinitionとかData Duplicatorとか言われるが、明確な定義がない

ブロック単位でデータをコピーする低レベルツール

dd if=/dev/zero of=helloos.img bs=1 count=0 seek=1474560
				    -> 0バイト書く -> ここに書き込み位置を移動


そもそもブロックサイズとは
ディスクの読み書きは「まとめて」行われる。

HDDやSSDは1バイト単位で読み書きができない。最小単位(ブロック)がある。

[メモリ]：１バイト単位でアクセス可能
[ディスク]：ブロック単位でしかアクセスできない。
[512バイト][512バイト]

1バイトだけ読みたくても、ブロック全体を読み込むしかない。

ディスクの物理ブロックサイズ

昔のHDD, フロッピー  512バイト(セクタ)
最近のHDD, SSD       4096(4K)
CD/DVD               2038バイト
ファイルシステム     4096バイト

このようにディスクによって変わる。

ddのbs(ブロックサイズ)は、物理ブロックとは別で、ddが一回のread/writeで扱うサイズを指定する。

bs=1      1バイトずつ read -> writeを繰り返す。(おそい)
bs=512    512バイトずつ
bs=4M     4MBずつ

bs=1が遅い理由
read(1), write(1)を繰り返す。
その分のシステムコールが毎回発生する。

dd bs=1   seek=1474560
dd bs=512 seek=2880

これが両方おんなじ意味らしい。

seekの計算方式

実際の移動バイト数 = bs * seek



#### そもそも!!!

seekとは。「読み書き位置の移動」

ファイルポインタの概念

ファイルを開くと、今どこを読み書きするのかを示すポインタが存在する。

helloos.img
[A][B][C][D][E][F][G]
-> ファイルポインタ(最初は先頭にある。)

read(3バイト) -> A, B, Cを読む
ファイルポインタが自動で進む。

またread(2)-> D, Eを進む


seekとは、「ファイルポインタをジャンプさせる命令」
fseek(fp, 5, SEEK_SET) => 先頭から5バイト目に移動

→ファイルポインタがジャンプする。

gcc -nostdlib のオプションでビルドしますがどういう意味だったのか。

OSが存在することを前提にした標準的なおまじないを一切つけない。という意味。

GCCが -nostdlib をつけなかった場合、裏側でやっていること。
gcc foo.c -o foo とするだけでGCCは裏側で大量のものを勝手にリンクしている。

```
crt1.o <- エントリポイント。mainを呼び出す前の準備
crti.o <- コンストラクタ初期化
[書いたソースコード]
crtin.o コンストラクタ後処理
+
libc.so <- printf, malloc, exitなどの標準C関数
libgcc.a <- GCCが必要とするランタイム補助(64bit除算など)
```

こんなのを色々くっつけてELFの実行ファイルを作る

crt1.oの中身(概念)

void _start(){
	// スタックセットアップ
	// 環境変数・引数の整理
	// グローバル変数の初期化
	main(argc, argv, envp);  // ここでmain()を呼ぶ
	exit(return_value);      // OSにプロセス終了を通知
}


ちなみに、リンカスクリプトの
OUTPUT_FORMAT("binary");

の"binary"は、ELF形式じゃなく生バイナリを出力せよ。というもの

通常は、ELFヘッダなどが大量にくっつく。
これが必要ないので、"binary"

GCCの関連オプション

-nostdlib      : スタートアップファイル＋標準ライブラリの両方外す
-nostartfiles  : crt1.o等のスタートアップファイルだけ外す
-nodefaultlibs : libc等の標準ライブラリだけ外す
-ffreestanding : 「OSなし環境」とコンパイラに伝える。(最適化の前提が変わるらしい)



### ipl hariboteの二つのバイナリを使ってFATを作る。


```
dd if=/dev/zero of=floppy.img bs=512 count=2880
dd if=ipl of=floppy.img bs=512 count=1 conv=notrunc
dd if=haribote of=floppy.img bs=512 seek=1 conv=notrunc
```

conv=notrunc とは

「truncate(切り詰め)」を理解する。

truncate = ファイルを書き込んだサイズで切り詰めること。
ファイルに書き込む時、OSのデフォルト動作は、
書き込むファイルに対して、3バイトだった時。
3バイトに縮む

これがtruncate(切り詰め)

conv=notruncって書く必要がある。

IPLからOSへのジャンプの仕組み

1. BIOSがIPLをロードするアドレス。

BIOSはブートセクタ(IPL)を物理アドレス0x7c00に自動ロードする。これは固定ルール。
なので、0x7c00にロードされたとしてその後のsp, osのロード場所を考える。

2. IPLがOSをロードするアドレス

movw $0x820, %ax
movw %ax, %es
movw $0, %bx
なので、

セクタ2: 0x7c00 + 0x0200 => 0x7e00
じゃないの？？

とは思うが, iplのプログラムをみると、0x0820と書いてある。
ということは、
0x7c00にiplをロードしているのて。0x7c00 - 0x7dffまでの512バイトはiplで使っている。それ以降の 0x7e00 - 0x081ff まではバッファとして開けてある。

floppy.imgをddコマンドで、無理やり作った場合、
OSのプログラムはiplのすぐ次の第二セクタに保存されているため、0x8200のアドレスにOSはロードされるが、FAT12の仕組みでいうと、jmp 0x820 となる。
